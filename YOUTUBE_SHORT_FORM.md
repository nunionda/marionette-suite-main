# 케이팝 아이돌 팬튜브 채널 콘텐츠 자동화 (에스파)

<aside>
📌

**문서 목적**

- 에스파 팬튜브(숏폼) 채널 운영을 *End-to-End 자동화*하기 위한 요구사항과 실행 계획을 한 문서로 정리한다.
- 2주 MVP 범위, 합격 기준, 운영·리스크 정책을 명확히 해서 구현과 운영이 바로 가능하도록 한다.

## 0. 한줄 요약

공식 소스 업로드를 감지하고 하이라이트 구간을 생성해, 컷 편집·자막·메타데이터를 자동 생성한 뒤 **검수(필수)**를 거쳐 YouTube Shorts에 예약 업로드하고 성과를 리포팅하는 자동화 시스템.

---

## 1. 채널/콘텐츠 전략(운영 관점)

### 1.1 컨셉

- 전 세계 K-POP 팬이 **공식 채널과 연계**해 가장 트렌디한 숏폼을 빠르게 소비할 수 있는 팬덤 중심 허브

### 1.2 목표 및 KPI

- **목표**: 팬덤 중심 숏폼 허브 구축 + 공식 채널 유입(전환) 유도
- **KPI**
    - 채널: 구독자, 조회수, Shorts 평균 시청지속시간(AVD), 시청 유지율(1초/3초), 저장·공유, 댓글 참여율
    - 전환: 공식 채널 클릭·유입(설명란/고정댓글 링크 기반)
    - 운영: 영상 1개 제작 리드타임, 1일 발행량, 오류율(업로드 실패/저작권 경고)
- **타깃**: 글로벌 팬 (EN/JP/ES 우선)

### 1.3 콘텐츠 기둥(Content Pillars)

- 컴백/무대 하이라이트: 방송·공연 클립 핵심 포인트 10–20초
- 멤버별 모먼트: 표정, 제스처, 밈 포인트
- 트렌드 리액션: 틱톡·유튜브 트렌드 사운드에 맞춘 편집
- 입문용 요약: “에스파 세계관/용어 30초 정리” 같은 지식형
- 팬 참여형: 댓글 질문·투표 기반 다음 편집 포인트 선정

## 2. 제품 요구사항(PRD)

### 2.1 제품 개요

- **제품명**: K-POP Idol FanTube Content Automation (Aespa)
- **문제 정의**
    - 핫한 구간을 빠르게 캐치하기 어렵다.
    - 반복 편집(컷·자막·캡션·포맷)이 시간이 많이 든다.
    - 멀티 플랫폼 업로드·해시태그·리포팅이 번거롭다.
- **핵심 가치**
    - 속도: 이슈 발생 → 업로드 리드타임 단축
    - 품질: 템플릿과 자막 품질의 일관성
    - 확장성: 다수 아티스트·플랫폼·다국어

### 2.2 범위 정의(MVP / V1 / 제외)

- **MVP(2주 내)**
    - 소스 감지: 지정 채널 신규 업로드 감지
    - 후보 선정: **룰 기반 + 운영자 타임스탬프 입력**으로 후보 구간 생성
    - 편집 자동화: 컷 분할 + 템플릿 적용 + 자동 자막(KR/EN)
    - 메타데이터 생성: 타이틀 3안 + 해시태그 생성 + 설명란 템플릿 적용
    - 배포: YouTube Shorts 예약 업로드
    - 리포팅: 일간 리포트(상위 영상, 유지율, 훅 성과)
    - 권리/크레딧: 고정 크레딧·면책 문구 자동 삽입
- **V1(4–6주)**
    - 멀티 플랫폼(틱톡/릴스) 동시 배포
    - A/B 테스트(타이틀·썸네일 규칙 기반)
    - 팬 참여형(댓글 질문 수집 → 다음 편집 포인트 반영)
    - 언어 확장(JP/ES)
- **명시적 제외(이번 범위 아님)**
    - 얼굴/인물 인식 기반 자동 하이라이트(고도화)
    - 완전 무검수 자동 업로드(품질·리스크상 제외)
    - 비공식/재업로드 채널 크롤링 등 불법 소스 수집

### 2.3 MVP 합격 기준(정량/정성)

- **처리/발행 성능**
    - 하루 **N개** 자동 렌더링 성공 (N은 운영 리소스에 맞춰 설정)
    - 업로드 파이프라인 실패율 ≤ **5%**
    - 원본 업로드 감지 → 업로드 예약까지 평균 리드타임 ≤ **60분**
- **품질(검수 기준 충족률)**
    - 자막 가독성(1–2줄 유지, 과밀 금지) 기준 충족률 ≥ **95%**
    - 크레딧/면책 문구 자동 삽입 누락 **0건**
- **성과(초기 2주 기준 최소치, 운영자가 확정 필요)**
    - Shorts 평균 AVD ≥ **X초**
    - 1초 유지율 ≥ **Y%**
    - 공식 링크 클릭 ≥ **Z회/주**

## 3. 기능 요구사항(상세)

### 3.1 소스 수집/권리/가이드

- 운영자가 **공식 채널/공식 SNS URL 목록**을 등록/수정할 수 있어야 한다.
- 각 소스별로 다음을 기록할 수 있어야 한다.
    - 사용 가능 범위(클립 길이/구간 제한 포함)
    - 크레딧 문구
    - 면책 문구
    - 리스크 등급
- 각 원본에 대해 **원본 URL과 식별자(videoId 등)**를 보관해야 한다.

### 3.2 후보 선정(하이라이트 추출)

- 입력(원본 영상)에서 **숏폼 후보 구간(10–20초)**을 최소 1개 이상 생성한다.
- MVP는 다음을 우선 지원한다.
    - **룰 기반**: 후렴/훅 타임스탬프 입력(운영자 입력 포함)
- 후보마다 다음을 저장한다.
    - 시작/종료 타임스탬프
    - ruleType
    - rationale(근거)

### 3.3 편집 자동화

- 템플릿(프리셋) 2종을 선택 적용할 수 있어야 한다.
- 자동 자막(KR/EN)을 생성해야 한다.
    - 싱크
    - 줄바꿈/속도/가독성
- 오프닝 0.5초, 엔딩 1초 CTA를 자동 삽입해야 한다.
- 고정 크레딧·면책 문구를 자동 삽입해야 한다.

### 3.4 타이틀/해시태그/설명 생성

- 타이틀 3안을 생성하고 운영자가 1개를 선택할 수 있어야 한다.
- 해시태그를 언어별로 생성해야 한다.
- 설명란 템플릿에 공식 링크/크레딧/면책 문구를 자동 적용해야 한다.

### 3.5 검수(필수 게이트) / 업로드 / 재시도

- 업로드 전 **검수 단계(필수)**를 제공해야 한다.
    - 검수 항목: 자막 오타, 화면 크롭, 첫 1초 훅, 권리/리스크 표시, 링크/UTM 적용
- 검수 결과는 다음 상태를 지원해야 한다.
    - **Approved(승인)**: 예약 업로드 진행
    - **Changes requested(수정 요청)**: 편집/메타데이터 재생성
    - **Rejected(반려)**: 폐기(자동 재시도 금지)
- YouTube Shorts **예약 업로드**를 지원해야 한다.
- 업로드 실패 시 재시도 및 실패 로그를 남겨야 한다.
    - 재시도 정책: 최대 N회, 지수 백오프
    - 중복 업로드 방지: idempotency key(원본ID+구간+템플릿버전)

### 3.6 성과 수집/리포트/추천

- 영상별 성과를 수집한다.
    - 조회수, AVD, 유지율, 좋아요/댓글/공유, 국가/언어(가능 범위)
- 일/주간 리포트를 생성한다.
- 상위 20% 영상 패턴(길이, 자막 속도, 훅)을 요약해 다음 제작에 반영할 수 있어야 한다.

## 4. 비기능 요구사항

- **신뢰성**: 실패 시 원인과 재시도 경로가 명확해야 한다.
- **확장성**: 아티스트/멤버 추가 시 설정만으로 확장 가능해야 한다.
- **보안**: API 키/토큰 안전 보관, 접근 권한 분리, 만료 알림
- **컴플라이언스**: 운영자 승인 없이 자동 발행하지 않는다.
- **관측성**: 단계별 로그(감지→후보→편집→검수→업로드→리포트)를 남긴다.

## 5. 정책 및 템플릿(권리/운영)

### 5.1 원칙

- 공식 소스만 사용한다.
- 검수 승인 없이는 발행하지 않는다.

### 5.2 크레딧/면책 문구 템플릿(초안)

- **크레딧(설명란/고정댓글 중 택1)**
    - `Source: Official aespa / SMTOWN channel. All rights belong to the original owners.`
- **면책(필요 시)**
    - `This is a fan-made highlight for commentary and discovery. Please visit the official channel for the full content.`

### 5.3 저작권 경고 발생 시 대응

1. 해당 소스/구간 제한 + 훅/자막/타이틀 변경
2. 자체 해설(텍스트/보이스) 강화 + 템플릿 변경
3. 반복 발생 시 운영 중단 + 재발 방지 룰 업데이트

## 7. 아키텍처 제안(MVP)

- **Orchestrator(워크플로우)**: n8n 또는 Make
- **Ingest(감지)**: YouTube Data API 폴링 또는 RSS + 키워드 필터
- **Job Queue(작업 큐)**: SQS/Redis Queue 또는 DB 테이블
- **Processing(편집)**
    - 다운로드/프록시 생성
    - 후보 구간 컷팅(FFmpeg)
    - 9:16 리프레임(초기: 센터 크롭 고정 + 안전영역 가이드)
    - 자막(Whisper) + 번역(DeepL/Google) + 스타일 적용
    - 오프닝/엔딩/크레딧 오버레이(FFmpeg drawtext + 템플릿)
- **Metadata**: LLM 프롬프트 템플릿 + 금칙어/브랜드 룰 후처리
- **Review UI(검수)**: Notion/웹 폼(승인 버튼) + 프리뷰 링크
- **Publish(업로드)**: YouTube API 예약 업로드 + 재시도 + 중복 방지
- **Analytics(성과)**: YouTube Analytics API + 일간 집계

## 8. 데이터 모델(최소)

- **Source**: id, type(youtube), channelId, url, creditText, disclaimerText, riskLevel
- **Asset(원본)**: id, sourceId, videoId, publishedAt, title, rawUrl
- **CandidateClip**: id, assetId, startSec, endSec, ruleType, rationale
- **RenderJob**: id, candidateClipId, templateId, langSet, status, errorCode, outputUrl, createdAt
- **ReviewDecision**: id, renderJobId, decision(approved/changes/rejected), reviewer, checklistResult, decidedAt
- **PublishJob**: id, renderJobId, platform, scheduledAt, status, uploadId, errorCode
- **DailyMetrics**: date, publishId, views, avd, retention1s, shares, comments, geo

### 8.1 로그 최소 필드(관측성)

- jobId
- stage(detect/candidate/render/review/publish/report)
- sourceId, assetId, candidateClipId, renderJobId
- templateVersion
- status, errorCode, errorMessage
- createdAt, updatedAt

## 9. 구축 계획(Implementation Plan)

<aside>
🛠️

**목표**

- 2주 내 *End-to-End 자동화 MVP*를 구축하고, 운영 루틴과 리스크 대응 체계를 함께 만든다.
</aside>

### 9.1 착수 전 의사결정(0.5일)

- 편집 체인: **FFmpeg 중심**으로 고정 (CapCut/Pr/AE는 V1 이후)
- 자막: Whisper(로컬/클라우드) 결정
- 번역: DeepL vs Google 결정
- 오케스트레이션: n8n/Make 선택

### 9.2 2주 스프린트 플랜

#### D1–D2: 기반/설계

- 요구사항 확정: MVP 합격 기준(KPI/품질/실패율)
- 데이터 모델 확정 + 작업 상태(state) 정의
- 크레딧/면책 템플릿 확정 + 금칙어 룰 정의
- 템플릿 v1 스펙 문서화(자막/오프닝/엔딩/크레딧)

#### D3–D5: 파이프라인 1차(감지→후보→렌더)

- 신규 업로드 감지(채널ID 기반) + 작업 큐 적재
- 후보 구간 생성(룰 기반 + 운영자 타임스탬프 입력)
- FFmpeg 컷팅 + 9:16 센터 크롭 + 오버레이
- Whisper 자막 + EN 번역 + 자막 스타일 적용

#### D6–D7: 검수→업로드→로그

- 검수 스텝 구현(승인/수정요청/반려, 체크리스트, 결정 로그)
- YouTube 예약 업로드 연결 + 재시도/중복 방지
- 에러 분류/알림(슬랙/이메일 중 1개)

#### W2: 리포트/개선 루프

- 일간 리포트 자동 생성(조회수/AVD/유지율/업로드 실패)
- 상위 20% 패턴 요약(길이/훅/자막 속도/템플릿)
- 다음 소재 추천 룰(원본/멤버별 백로그 우선순위)

### 9.3 Go/No-Go 체크

- 하루 N개 자동 렌더링 성공(에러율 ≤ 목표)
- 승인 후 예약 업로드 정상 동작
- 업로드/리포트 로그 누락 없음
- 크레딧/면책 문구 자동 삽입 100% 준수

    - [ ]  **리스크 대응 체크리스트 확정**: 저작권 경고 플로우, 악성댓글 가이드, 업로드 실패 재시도/로그 기준
- **P1(병렬로 진행 가능, 개발 속도/품질에 직접 영향)**
    - [ ]  **콘텐츠 기둥별 템플릿 2종 제작**: 자막 스타일, 오프닝 0.5초/엔딩 1초 CTA, 고정 크레딧 자동 삽입
    - [ ]  **채널 목표·KPI 확정**: KPI 정의 + 2주 목표치(X/Y/Z) 설정
    - [ ]  **2주 실행 체크리스트 담당자 지정**: D1–D2 / D3–D5 / D6–D7 / W2 산출물과 담당자 지정
- **P2(구축 후에도 개선 가능, 하지만 방향성은 있어야 함)**
    - [ ]  **성과 수집/리포트 포맷 정의**: 일간 지표, 상위 20% 패턴 분석 항목, 다음 소재 추천 규칙

### 10.2 개발 착수 전 ‘최소 확정 세트’(이 6개는 꼭 잠그기)

- [ ]  **권리/운영 원칙**: “공식 소스만”, “검수 승인 없이는 발행 금지” 최종 확인
- [ ]  **YouTube Shorts 업로드 정책**: 예약 업로드 가능 여부, 계정/채널 권한, API 사용 범위 확인
- [ ]  **검수 체크리스트 고정**: 자막 오타, 화면 크롭, 첫 1초 훅, 권리/리스크 표시, 링크/UTM 적용
- [ ]  **실패/재시도 정책**: 최대 N회, 지수 백오프, 중복 업로드 방지 키 정의
- [ ]  **로그 최소 스키마**: stage/status/errorCode/errorMessage/id 매핑(Asset/Candidate/Render/Publish) 고정
- [ ]  **MVP 합격 기준 숫자**: N/X/Y/Z 값(또는 범위) 확정

### 10.3 최종확인(Go/No-Go)

- **지금 상태 기준으로는 ‘조건부 Go’**: 아래 P0 항목을 확정/서명하면 개발 착수해도 된다.
- **No-Go 조건(착수 보류 사유)**
    - 권리 범위/크레딧/면책 문구가 확정되지 않아, 자동 삽입 누락 시 리스크가 큰 경우
    - 검수 게이트를 우회할 수 있는 운영/시스템 경로가 남아있는 경우
    - 업로드 중복 방지 키와 재시도 정책이 없어 채널 스팸/중복 업로드 위험이 있는 경우
    - 로그/관측성이 없어 실패 원인 추적이 불가능한 경우
